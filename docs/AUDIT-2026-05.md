## Blocker fixed — Savings goals (commit `fix(savings)`)

Root cause: the feature was half-built. `recalculateGoalProgress()` was defined but
**never called**, and `transactions.savings_goal_id` was **never set** by any UI — so a
goal's `current_amount` never reflected money actually moved into savings. This is the
"separo dinero a un saving pero no se relacionan" symptom.

Chosen model: **account-linked derivation (cannot drift).** When a goal is linked to an
account, `current_amount` / `progress` / `is_completed` are computed live in
`getSavingsGoals` from that account's real balance (initial opening + non-deleted
movements). Moving money into the savings account now updates the goal automatically with
zero manual tagging. The goal currency is locked to the account currency in `GoalDialog`.
Goals without an account keep an optional, manually-entered current amount (new field).
Removed the dead `recalculateGoalProgress`.

Files: `src/actions/savings-goals.ts`, `src/app/(dashboard)/savings/_components/GoalDialog.tsx`,
`src/lib/validations/savings-goals.schema.ts`.

How to use: create a savings account, create a goal and pick that account → the card
fills as you transfer money in. Limitation: 1 account ↔ 1 goal (multiple goals on one
account would each show the full balance); for sub-goals use a dedicated account or a
manual goal.

## Other bug fixed — Next-month preview (commit `fix(months)`)

`previewNextMonthFromLatest` summed transaction legs **without** the `deleted_at` filter
that `createMonth` and `recalculateOpeningBalances` both apply, so the projected opening
balances counted soft-deleted transactions. Added the filter.
